## 一、技术简历的黄金公式

### STAR-Q：每个项目段落都要有这五块

```
背景(为什么做) + 技术方案(做了什么·选型理由) + 你的角色(独立做了哪部分)
+ 量化结果(提升多少) + 一句话亮点(和别人不一样在哪)
```

### 三条铁律

1. **动词开头 + 量化结尾**
   - ❌ "负责 RAG 系统"
   - ✅ "设计三路混合召回 + RRF + Reranker，将 Recall@5 从 X% 提升到 Y%"

2. **技术名词要成体系，不堆砌**
   面试官看到 `Qwen3-Embedding + BGE-M3 + BM25 + RRF + BGE-Reranker` 一串，会默认你懂检索链路——**但每个词你都要能被追问到底**。堆你答不出的词，是给自己挖坑。

3. **暴露一个"取舍"比堆十个技术更显水平**
   一句"用存储换检索质量（MongoDB 存父块、Milvus 只存向量）"，胜过十个名词。

## 二、你的两个核心项目怎么改

### 项目 1：Tesla RAG（你最能打的项目，排第一）

**改前**（`introduce.md` 原文）：
> "构建一个基于 Tesla 汽车用户手册的车书问答系统……提升答案生成的准确性、快速精准的回答用户汽车相关问题。"

**问题**：全是"做了什么"，**没有一个数字**；"提升准确性、快速精准"是空话。

**改后模板**（把 `【】` 用你的真实实验数据填上）：
> 独立构建 Tesla 用户手册 RAG 问答系统，支持答案溯源（页码 + 关联插图）。
> - **解析/切分**：PyMuPDF 抽取图文 + 元数据；滑动窗口 / 父子文档 / 语义切分三策略，解决"操作步骤跨页被切碎"问题；
> - **检索**：Dense(Qwen3-Embedding) + Sparse(BGE-M3) + BM25 三路混合召回，RRF 粗排 + BGE-Reranker 精排，相比单路召回 **Recall@5 +【X】%**；
> - **溯源**：基于建库元数据实现页码引用与插图关联，**faithfulness 达【Y】%**。

> ⚠️ **应届生最大的坑**：没有量化 = 项目被当 demo。如果你还没跑消融实验（单路 vs 三路、有无 Reranker），这周就补上——这是你简历可信度的地基。

### 项目 2：Agent Skill Contract（研究项目，排第二）

**一句话亮点（背下来）**：
> "本质是把软件工程的**契约式设计（Design by Contract）搬到 LLM Agent 上**。"

**注意**：这是研究项目，面试官默认有导师。**别假装全独立**——讲清"我具体写了哪些代码、做了哪些决策"（契约结构设计、AST 静态分析抽签名、运行时验证器原型）。诚实反而加分。

### 排版原则
- 应届简历**一页纸**；项目按"最能打的排最前"：**Tesla RAG > Agent Contract > AI for SE**。
- 技能栏别写"了解 XX"，写"熟悉/掌握 XX（做过 YY）"，每个技能都能对应到一个项目。

## 三、模拟面试：你会被这样追问

> 完整题库在 `qa.md`。这里是你**必须练到脱口而出**的核心题。答题总原则：
> **结论先行 → 举一个具体例子 → 说清 trade-off → 主动暴露局限。**

### Tesla RAG 必练题（几乎必问）

| 题 | 你要答到的深度 | 常见追问 |
|---|---|---|
| **一分钟讲清系统** | 用户问操作→检索手册→LLM 生成→给页码和插图。难点在图文长 PDF | "为什么不直接丢给长上下文大模型？"（答：成本/精度/可溯源三点）|
| **完整 RAG 链路** | 30 秒不看稿画出：PDF→解析→三种切分→存储→三路召回→RRF→Reranker→生成 | 让你**当场画** |
| **为什么三路召回** | Dense 懂语义换说法、BM25 精确术语、Sparse 折中；三路互补 | **"BGE-M3 你用它哪部分？"**（答 sparse 那一路，答不出=只会名词）|
| **RRF 是什么** | 只看排名不看分数，`1/(k+rank)` 求和，规避不同尺度归一化问题 | "k 取多少？"（常用 60）|
| **为什么还要 Reranker** | 召回是 bi-encoder 快但糙做粗筛；Reranker 是 cross-encoder 慢但准做精排 | **"bi-encoder vs cross-encoder 区别？"**（分水岭题）|
| **微调做了什么** | 如实答！讲清 RAG 解决知识来源、微调解决风格/领域适配 | "微调 vs 只做 RAG 增益多大？值不值？"（**最容易翻车，别夸大**）|

### Agent Contract 必练题

| 题 | 你要答到的深度 | 追问 |
|---|---|---|
| **一分钟讲清** | Agent 用工具缺"何时调/参数怎么填/调完对不对"，我给每个 Skill 生成结构化契约 | "这不就是 function calling 的 JSON Schema 吗？"（答：Schema 只管参数格式，我管行为层+能自动生成）|
| **静态分析抽了什么** | Python `ast` 抽签名/类型注解/默认值/异常分支/入口校验逻辑 | "只抽签名怎么知道行为？"（答：签名+docstring+异常覆盖接口契约，深层行为靠 LLM+运行时补）|
| **运行时怎么验证** | 调用前查 precondition（不满足直接拦截省一次调用），调用后查 postcondition | "调用前拦截会误杀吗？"（答：硬拦截只用高置信约束，低置信只警告）|
| **怎么衡量效果** | 生成质量（准确率/召回率 vs 金标准）+ 端到端（误调用率↓、任务成功率↑）| "baseline 是什么？"（答：Agent 只看原始自然语言描述）|

### 压力题（随机抽，考思维不是记忆）
- "用户问的手册里根本没有怎么办？" → prompt 约束"无依据就说不知道" + 检索分数阈值兜底。
- "你的方法 scale 到上千个 Skill 还成立吗？" → 生成离线可缓存、运行时分层校验硬约束零成本，瓶颈在语义层 LLM 调用。
- "为什么不用形式化验证？" → LLM 行为不确定无法完全形式化，追求工程够用而非数学完备，是 trade-off。

## 四、两个应届生通病（面试官一眼看穿）

1. **不敢暴露局限**——面试官最烦"我这个项目没有缺点"。主动说 1–2 个真实局限（如"多 Skill 已执行几步才失败，有副作用的回退还没做好"）反而让人信任。
2. **只堆名词不讲取舍**——被问"为什么用它"时答不出"不用会怎样"。每个技术选型都要能回答**"为什么用它、不用会怎样"**。

> **万能锚点技巧**：`qa.md` 里"PDF 解析"这个例子贯穿契约项目全程。给自己每个技术答案都准备**一个具体例子**，好记好讲，面试时不会空。

### 投递优先级
1. **主攻岗批量投**：AI 应用开发 / 大模型应用 / Agent 开发 / RAG 工程师。
2. **辅攻岗补充**：NLP 应用算法（算法够硬时冲）、AI 产品/解决方案（兜底）。
3. **不主投**：纯后端（元宝后台这类）、Agent 后训练算法（社招门槛，应届够不到）。
4. **渠道优先级**：内推 > 官网提前批 > 牛客/BOSS。**内推能跳过简历筛，务必攒内推码。**
