# 面试深挖问题清单 · Agent Skill 行为契约项目

> **用途**：第 5 节课模拟面试用。面试官（老师）按顺序问，学员先答，老师用「追问」施压，再对照「参考答案」校准。
> **重要说明**：项目仍在进行中，很多**指标/实现细节需要学员用真实数据替换**（文中标 `【学员填真实值】`）。参考答案给的是"应该答到什么深度"的模板，不是让她背。
> **答题总原则**：结论先行 → 举一个具体例子 → 说清 trade-off → 主动暴露局限（面试官最烦"没有缺点"）。

## 一、项目深挖（考经历真实性 + 思考深度）

### Q1. 用一分钟讲清楚这个项目在做什么、为什么做

**参考答案：**
> "现在的 Agent 用技能（Skill/工具）时，技能大多只有一段自然语言描述，导致三个问题：Agent 不确定**什么时候该调**、**参数怎么填**、**调完对不对**。我做的是给每个 Skill 生成一份结构化的『契约』——把前置条件、输入输出约束、后置验证、失败恢复显式写出来，让 Agent 在选技能、填参数、组合、执行时主动读契约来约束自己的行为。核心是把 Agent 对技能的『隐式理解』变成『显式、可校验的接口约束』。本质是把软件工程的**契约式设计（Design by Contract）搬到 LLM Agent 上**。"

**追问**：这不就是 function calling 的 JSON Schema 吗？有什么区别？
**答**：JSON Schema 只约束**参数的类型/格式**（静态、语法层）；我的契约还包含**使用条件（何时可调）、后置语义验证（结果对不对）、失败恢复**——是**行为层**的约束，且能**自动生成**，不用人手写。

**提示**：这题决定第一印象，务必练到脱口而出、40 秒内讲完。

---

### Q2. 这个项目里，哪些是你独立做的？团队做了什么？

**参考答案（诚实但突出个人贡献）：**
> "整体研究方向和框架由导师把关。我独立负责了：① 契约结构的设计（四类字段的定义）；② 自动生成 pipeline 的实现——包括用 AST 做静态分析抽签名、用 LLM 抽自然语言约束、以及两者的融合；③ 运行时验证器的原型。评测方案和导师一起定的。"

**追问**：如果是你一个人，最难自己啃下来的是哪部分？
**答**：`【学员按真实情况答，见 Q3】`

**提示**：应届研究项目，面试官默认有导师指导，**不要假装全独立**（会被追问穿），关键是把"我具体写了哪些代码/做了哪些决策"讲清楚。

---

### Q3. 整个项目最大的技术挑战是什么？怎么解决的？

**参考答案（选一个真实的深讲，推荐"多源证据冲突"）：**
> "最大挑战是**多源证据不一致**。同一个 Skill，文档里写的行为、源码实际做的、脚本里的边界条件经常对不上——比如文档说『支持 PDF 和图片』，但代码里只处理了 PDF。如果直接信文档生成契约，运行时就会误判。
> 我的解法是给证据**分优先级**：涉及**类型/参数结构**的以**源码静态分析为准**（代码不会骗人）；涉及**使用意图/语义约束**的以**文档为主、代码佐证**；冲突时标记为『低置信』并保留两种来源，交给运行时用实际执行结果去消解。"

**追问**：那如果代码是动态的、AST 也分析不出来呢？
**答**：见 Q8（降级为运行时校验 + LLM 兜底）。

---

### Q4. 你怎么衡量这个方法有没有效果？用什么指标？

**参考答案（这是应届研究最容易被问穿的地方，必须准备量化）：**
> 分两层评估：
> - **生成质量**：契约条款的**准确率/召回率**——生成的约束和人工标注的『金标准契约』比对（准确率 `【填真实值，如 ~85%】`）。
> - **端到端效果**：接入 Agent 后，对比有契约 / 无契约两组，看 **① 工具误调用率下降多少、② 任务成功率提升多少、③ 无效调用（参数不合法就发起）被拦截多少**。`【填真实值】`

**追问**：baseline 是什么？数据集哪来的？
**答**：baseline 是"Agent 只看原始自然语言描述"。数据集用 `【真实来源，如：公开的 MCP 工具集 / 自建的 N 个 Skill】`，人工标注金标准契约。

**提示**：**没有量化 = 项目不可信**。哪怕是小规模实验，也要给出具体数字。上课前必须让她把真实数据整理出来。

---

### Q5. 如果现在重做，你会怎么改进？

**参考答案（展示反思能力，加分项）：**
> "三点：① 自动生成现在还依赖人工校对，想引入**自动一致性检查**减少人工；② 后置验证目前对『语义是否正确』判断偏弱，想结合**执行结果 + LLM-as-judge** 做更细的语义校验；③ 想扩到**多 Agent 协作**场景，用契约来协调不同 Agent 之间的接口。"

**提示**：这题一定要准备，答不出来 = 没深入思考过。

---

### Q6. 这个方向学术界/工业界有相关工作吗？你的创新点在哪？

**参考答案：**
> "相关的有：软件工程里的 Design by Contract、形式化规约；LLM 领域有工具学习（Tool Learning）、function calling 的 schema 约束、以及一些 Agent 可靠性/约束的工作。
> 我的差异点是：**把契约的『自动生成』和『运行时验证』连成闭环**——不是人手写契约，而是从文档+代码多源自动抽；而且契约不只约束参数，还约束**行为和使用时机**，Agent 能在运行时动态读取和推理。"

**提示**：能说出几个相关领域名词就显得有 sense，不用背具体论文（除非她真读过）。

## 二、技术深挖（考硬功底）

### Q7. Contract 具体用什么格式表示？为什么选它？

**参考答案：**
> "用 **结构化 schema**，实现上基于 `【JSON Schema / Pydantic 模型，填真实的】`。选它的原因：① **机器可执行**——运行时能直接拿来校验；② 生态成熟，和 function calling、MCP 的工具描述天然兼容，容易接入现有 Agent 框架；③ 可读可扩展。
> 一份契约大概长这样（举例）：
> ```json
> {
>   "skill": "parse_pdf",
>   "precondition": ["input.path 存在且后缀为 .pdf", "文件大小 < 50MB"],
>   "input": {"path": {"type": "string", "format": "filepath"}},
>   "output": {"type": "object", "required": ["pages", "text"]},
>   "postcondition": ["返回 pages > 0", "text 非空"],
>   "on_failure": ["重试 1 次", "降级为 OCR", "报错并跳过"]
> }
> ```"

**追问**：为什么不用自然语言 + LLM 直接判断，非要结构化？
**答**：自然语言判断**不稳定、不可复现、还费 token**；结构化的硬约束（类型、必填字段）能用**廉价的确定性代码**先过一遍，只有语义层才交给 LLM，**又快又可靠**。

---

### Q8. 静态程序分析这块，你具体抽了什么？怎么实现？动态代码怎么办？

**参考答案：**
> "用 Python 的 **`ast` 模块**解析源码，抽取：**函数签名、参数名和类型注解、默认值、返回类型、docstring、显式的 raise/异常、以及入口处的参数校验逻辑**（比如 `if not path.endswith('.pdf'): raise`——这类就是天然的前置条件）。
> **动态代码**（反射、运行时才确定的类型、外部调用）AST 抽不全，我的处理是**降级**：静态抽不到的约束，标记为『需运行时确认』，交给运行时验证器在实际执行时观测输入输出来补全，必要时用 LLM 读代码语义兜底。"

**追问**：只抽签名，怎么知道函数"行为"？
**答**：签名 + docstring + 异常分支能覆盖大部分**接口契约**；深层行为（副作用、状态改变）确实是静态分析的边界，这也是我为什么要**融合 LLM 语义理解 + 运行时观测**——三者互补。

**提示**：这题最能体现工程功底，`ast`、类型注解、异常分支这些词要说得出来。

---

### Q9. 用 LLM 从文档抽约束，怎么保证抽得对？幻觉了怎么办？

**参考答案：**
> "三层防幻觉：① **结构化输出**——让 LLM 按固定 schema 返回，不让它自由发挥；② **要求给证据**——每条抽出的约束要标注『来自文档哪一句 / 代码哪一行』，抽不出出处的丢弃；③ **交叉验证**——LLM 抽的约束要能被静态分析或运行时执行**印证**，印证不了的降为低置信、不进硬约束。
> 核心思想是：**LLM 只做『提议』，最终采纳要靠代码证据背书**，不让它单独决定契约。"

**追问**：那纯语义的约束（代码里印证不了的）你怎么办？
**答**：保留为"软约束/建议"，运行时用来做提示而非硬拦截，避免误报。

---

### Q10. 运行时怎么做验证？precondition 和 postcondition 分别在什么时候查？性能开销大吗？

**参考答案：**
> "在 Skill 执行外面包一层**验证器（守卫/wrapper）**：
> - **调用前**查 precondition + 输入约束——不满足**直接拦截**，根本不发起调用（省掉一次无效执行）；
> - **调用后**查 postcondition——不满足则触发失败恢复。
> **开销**：结构化的硬约束（类型、必填、范围）是**纯确定性代码判断，几乎零成本**；只有语义层的后置验证才可能调一次 LLM，这部分才有开销。所以我把验证**分层**——能用代码判的绝不用 LLM，把 LLM 调用降到最少。"

**追问**：调用前拦截，会不会误杀本来能成功的调用？
**答**：会有误报风险，所以硬拦截只用于**高置信约束**（源码印证过的），低置信的只警告不拦截——这是 precision/recall 的取舍。

---

### Q11. 后置验证最难的是"结果语义对不对"，LLM 输出又不确定，你怎么验证？

**参考答案（这是项目最硬核的点之一）：**
> "分三档：
> - **结构层**（有没有必填字段、类型对不对）→ 确定性代码校验；
> - **约束层**（数值范围、格式、非空）→ 规则校验；
> - **语义层**（答案是否真的回答了问题、有没有跳出边界）→ 这层没法用规则，我用 **LLM-as-judge + 执行结果比对**，让一个校验 LLM 按 postcondition 逐条判断。
> 关键是**不把所有验证都甩给 LLM**——大部分错误在前两档就拦住了，语义层只兜底，既准又省。"

**追问**：LLM-as-judge 本身也会错，怎么办？
**答**：对高风险场景可以多次采样投票 / 给 judge 也定 rubric；并且 judge 的结论是"建议"，最终可以结合执行是否报错等硬信号综合判断。

---

### Q12. 多个 Skill 组合时，契约怎么保证它们能接上？

**参考答案：**
> "把它当成**类型匹配 / 接口兼容**问题：Skill A 的 **output 契约**要能满足 Skill B 的 **input precondition**。组合前先做一次**契约层的静态检查**——A 的输出字段和类型能不能填进 B 的输入约束，不兼容就提前报错或插入一个适配步骤，而不是等运行到一半才崩。
> 好处是把很多组合错误**从运行时提前到规划时**发现，类似编译期检查。"

**追问**：中间结果不符合预期怎么动态调整？
**答**：运行时每步 postcondition 通过才进下一步；不通过就触发恢复策略（重试/换 Skill/回退），也就是简历里说的『根据中间结果动态调整执行策略』。

---

### Q13. 失败恢复机制具体怎么设计？

**参考答案：**
> "契约里给每个 Skill 定义 `on_failure` 策略，常见几档：**重试**（瞬时错误）→ **降级**（换一个更稳的替代 Skill 或退化方案，比如 PDF 解析失败降级为 OCR）→ **回退**（撤销已做的、回到上一个安全状态）→ **上报/跳过**。运行时验证器捕获到 postcondition 失败或异常，就按契约里定义的策略链依次尝试。"

**追问**：多 Skill 已经执行了几步才失败，怎么回退副作用？
**答**：这是难点，纯读操作好回退，有副作用的（写库、发消息）需要**补偿操作/幂等设计**——诚实说这是当前局限，`【学员按真实进展答】`。

---

### Q14. 这套东西和 MCP / Function Calling / Claude Code 的 Skills 是什么关系？

**参考答案（体现她懂生态）：**
> "它们是**互补**的。MCP 和 function calling 提供的是**工具的静态 schema（参数格式）**，Claude Code 的 Skills 是**自然语言描述的能力包**——但都缺『使用条件、后置验证、失败恢复』这些行为约束。我的契约相当于在这些标准之上**加了一层行为契约**，而且是**自动从文档和代码生成的**，可以直接增强现有的 MCP/Skill 生态，而不是另起炉灶。"

**提示**：能主动把项目和**主流工业标准**挂钩，是很强的加分信号，说明她不是闭门造车。

## 一、项目深挖（考经历真实性 + 系统设计能力）

### Q1. 一分钟讲清楚这个系统在做什么、解决什么问题

**参考答案：**
> "它是一个基于 Tesla 用户手册的问答系统——用户问『怎么开启哨兵模式』这类问题，系统从手册里检索出相关内容，用 LLM 生成答案，并且**给出引用的页码和相关插图**。
> 难点在于：汽车手册是**图文混排的长 PDF**，直接切块会丢上下文、丢表格和图。所以我在**解析、切分、检索、生成**四个环节都做了针对性优化：解析用 PyMuPDF + 多种切分策略保上下文；检索用**三路混合召回 + RRF 粗排 + Reranker 精排**保准确率；生成时带页码和插图引用保**可溯源**。整体目标是**又快又准、答案可信**。"

**追问**：为什么不直接把手册丢给一个长上下文大模型？
**答**：① 手册几百页，超上下文且**每次全塞成本高、延迟大**；② 长上下文里"大海捞针"精度会掉；③ 无法给出**精确页码溯源**。RAG 是**成本、精度、可溯源**三者的更优解。

---

### Q2. 这是个人项目，整个系统都是你一个人搭的吗？

**参考答案：**
> "是的，从 PDF 解析、数据入库、检索链路到问答接口都是我独立完成的。技术选型上参考了业界成熟方案（比如多路召回 + RRF + Reranker 是当前 RAG 的主流最佳实践），但每一步在这个手册场景下的**调参和适配是我自己做的**。"

**提示**：个人项目=全独立，反而要防"是不是跟着教程抄的"这种质疑——**用 Q3 的踩坑经历证明你真踩过坑**。

---

### Q3. 做这个项目最大的坑 / 挑战是什么？

**参考答案（选一个真实的深讲，推荐"切分策略"或"图文关联"）：**
> "最大的坑是**切分**。汽车手册一个操作步骤经常跨页、还带图表，用固定长度切块会把一个完整操作切碎，检索出来的片段答不全。
> 我的解法是**组合三种切分**：① **滑动窗口**保证块之间有重叠、不丢边界信息；② **父子文档**——用小块（子块）做精准检索、但返回给 LLM 时带上它所属的大块（父块）补全上下文；③ **语义切分**在语义边界处断开，避免把一句话劈两半。三者结合后，答案完整度明显提升。`【填真实对比数据】`"

**追问**：三种切分同时用，块会不会爆炸、冗余很大？
**答**：会增加存储和索引量，所以我在 MongoDB 存全量文本块+元数据、Milvus 只存向量，检索用子块、返回用父块，**用存储换检索质量**——这是有意识的 trade-off。

---

### Q4. 你怎么评估这个系统好不好？

**参考答案（RAG 评估要分两段，这是硬功底）：**
> "分**检索**和**生成**两段评估：
> - **检索质量**：Recall@k、命中率——问题对应的正确片段有没有被召回到 top-k。`【填真实值】`
> - **生成质量**：答案**准确率**（人工/LLM 打分）、**忠实度 faithfulness**（有没有幻觉、是不是基于检索内容答的）、**引用正确率**（页码/插图对不对）。
> - **对比实验**：单路召回 vs 三路混合、有无 Reranker——证明每个组件的增益。`【填真实值】`"

**追问**：没有标注数据集怎么评？
**答**：自己构造一批 QA 对（从手册里出题+标准答案），或用 **LLM 生成评测集 + 人工抽检**；faithfulness 可以用 RAGAS 这类框架或 LLM-as-judge。

**提示**：**能说出 Recall@k、faithfulness、RRF 增益对比**，直接区分"真做过 RAG"和"只会背流程"。

## 二、技术深挖（考 RAG 硬功底，重点区）

### Q6. 完整讲一下你的 RAG 链路，从 PDF 到答案

**参考答案（一句话串起全流程，先给全景）：**
> "**离线建库**：PyMuPDF 解析 PDF → 滑动窗口/父子/语义三种切分 → 文本块+元数据存 MongoDB → 向量化后存 Milvus。
> **在线问答**：用户 query →**三路召回**（Dense: Qwen3-Embedding / Sparse: BGE-M3 / 字面: BM25）→ **RRF 融合做粗排** → **BGE-Reranker 精排** top-k → 拼 prompt 交 LLM 生成 → 输出答案 + 页码 + 插图。"

**提示**：这题是总纲，务必能**不看稿、30 秒画出整条链路**。后面每个组件都是从这里展开。

---

### Q7. PDF 解析为什么用 PyMuPDF？图文混排怎么处理？

**参考答案：**
> "PyMuPDF（fitz）**速度快、能同时拿文本、坐标位置和图片**，适合手册这种图文混排。解析时我不只抽文本，还抽**每块文本的页码、bbox 坐标、以及页面上的图片**——把文本块和它附近的插图**在元数据里关联起来**，这样最后答案才能带出『相关插图』。表格这类结构化内容单独处理，避免被当普通文本切碎。"

**追问**：扫描版 PDF（图片型）怎么办？
**答**：PyMuPDF 抽不到文本层的就要上 **OCR**（如 PaddleOCR）兜底——这是当前方案的边界。

---

### Q8. 三种切分（滑动窗口/父子/语义）分别解决什么？为什么要三个一起用？

**参考答案：**
> - **滑动窗口**：块之间**重叠**，防止关键信息正好落在切割边界被割裂；
> - **父子文档**：**子块（小、精准）用于检索**，命中后返回**父块（大、上下文全）给 LLM**——兼顾『检索精度』和『上下文完整』；
> - **语义切分**：在**语义/段落边界**处断，而不是死板按字数，避免把一个完整语义切两半。
> 单用任何一种都有短板，组合起来在『不丢上下文』和『检索精度』之间取到更好的平衡。"

**追问**：父子文档具体怎么存、怎么关联？
**答**：每个子块存一个 `parent_id` 指向父块，检索命中子块后按 `parent_id` 去 MongoDB 取父块内容。

---

### Q9. 为什么用 MongoDB 存文本块？存了哪些东西？

**参考答案：**
> "MongoDB 是**文档型数据库**，天然适合存**结构不固定的文本块+元数据**。每条存：块内容、页码、bbox 坐标、父块 id、关联图片路径、切分类型等。
> **分工**：MongoDB 存**全量原文和元数据**（负责『取内容』），Milvus 存**向量**（负责『按语义找』）——检索时 Milvus 给出命中的块 id，再回 MongoDB 拿完整内容和元数据。这样职责清晰，也方便加页码/插图溯源。"

---

### Q10.【重点】为什么要三路召回？Dense、Sparse、BM25 各擅长什么？

**参考答案（这是本项目最核心的技术亮点）：**
> "因为**单一召回都有盲区**：
> - **Dense 语义召回（Qwen3-Embedding）**：懂『语义相似』，用户换个说法也能召回——但对**精确术语、型号、专有名词**不敏感；
> - **字面召回 BM25**：擅长**精确关键词匹配**（比如具体的按钮名、错误码），但**不懂同义**；
> - **Sparse 召回（BGE-M3）**：介于两者之间，稀疏表示对**关键词权重**敏感，兼顾一部分语义。
> 汽车手册里既有口语化提问、又有大量精确术语，**三路互补**才能既召得全又召得准，比任何单路都强。"

**追问**：BGE-M3 不是能同时出 dense/sparse/multi-vector 吗？你用它哪部分？
**答**：对，BGE-M3 是多功能 embedding，这里主要用它的 **sparse（词权重）表示**这一路；dense 那路我用 Qwen3-Embedding。（**这个细节答得出=真用过，答不出=只会名词**）

**提示**：这题几乎必问，务必把"每一路补什么盲区"讲到具体例子。

---

### Q11. 三路召回结果怎么合并？RRF 是什么、为什么用它？

**参考答案：**
> "用 **RRF（Reciprocal Rank Fusion，倒数排名融合）** 做粗排。它**只看每个文档在各路结果里的排名**，公式是每路 `1/(k + rank)` 求和，排名越靠前贡献越大。
> 为什么用它：三路召回的**分数尺度完全不同**（向量相似度 vs BM25 分），没法直接加；RRF **只用排名、不用原始分数**，天然规避了归一化问题，简单又稳，是多路融合的业界标配。"

**追问**：RRF 里 k 取多少？影响是什么？
**答**：常用 60。k 越大，各路排名差异被**平滑**得越厉害（更均衡）；k 越小，越突出各路的头部结果。`【学员填真实取值】`

---

### Q12. 已经有 RRF 粗排了，为什么还要 BGE-Reranker 精排？

**参考答案（考对『召回 vs 排序』的理解）：**
> "召回阶段的模型（embedding/BM25）是 **bi-encoder / 词匹配**——query 和文档**分开编码**，快但精度有限，适合从海量里**粗筛**出 top-N。
> **Reranker 是 cross-encoder**——把 query 和文档**拼在一起送进模型**，能建模两者细粒度的交互，**精度高但慢**，只能对少量候选算。
> 所以是**两阶段**：先用便宜的召回从全库粗筛几十条，再用贵的 Reranker 把这几十条**精排**出最相关的 top-k 给 LLM。**用速度换范围、用精度换质量**，兼顾又快又准。"

**提示**：能讲清 **bi-encoder vs cross-encoder** 的区别，是 RAG 进阶的分水岭，重点练。

---

### Q13. 简历提到"大模型微调和提示工程"，具体微调了什么？

**参考答案（这里要诚实，别夸大）：**
> `【学员按真实情况答，两种可能：】`
> - 若微调的是 **embedding/reranker**：用手册领域的 QA 对做了**领域微调**，让检索更贴合汽车术语；
> - 若微调的是**生成 LLM**：用 QA 数据做了 **LoRA 微调**，让答案风格更贴合手册、减少跑题；
> - **提示工程**：设计了 prompt 模板，强约束『**只基于检索内容作答、必须给出页码、不知道就说不知道**』来降幻觉。

| 对比维度 | 微调 embedding/reranker | 微调生成 LLM |
|---|---|---|
| 改的是哪一步 | 检索(找内容) | 生成(写答案) |
| 解决的问题 | 找得不够准、不懂专业术语 | 答得跑题、风格不对、爱脑补 |
| 用同样的 QA 数据学什么 | 问题↔正确段落的匹配关系 | 如何依据内容组织出好答案 |
| 常用方法 | 对比学习(拉近正样本) | LoRA 等轻量微调 |


**追问**：微调 vs 只做 prompt/RAG，增益有多大？值不值？
**答**：`【填真实对比】`。要点：**RAG 解决『知识来源』，微调解决『风格/格式/领域适配』**，如果只是问答，很多时候好的 RAG+prompt 就够，微调是锦上添花——**能说清这个边界=有判断力**。

**提示**：⚠️ 这是**最容易被问穿的点**。如果微调其实做得很浅，**别硬吹**，如实说"主要靠 RAG + prompt，微调做了小规模尝试"反而更可信。

### Q14. 页码引用和关联插图是怎么实现的？

**参考答案：**
> "全靠**建库时存的元数据**：每个文本块都带**页码、bbox、关联图片路径**（Q7/Q9 里存的）。检索命中某个块后，生成答案时把它的**页码一起带出**；插图是解析时按『图片在页面上和文本块的位置邻近关系』关联好的，答案引用了哪个块，就一并返回它关联的图。这样答案**可溯源、可视化**，用户能直接翻到手册对应页。"

## 四、给学员的答题提示（Tesla 项目专属）

1. **必背全景链路（Q6）**——30 秒画出 PDF→答案整条流水线，这是所有深挖的地基。
2. **三路召回（Q10）+ RRF（Q11）+ Reranker（Q12）是三连核心**，几乎必问，练到能讲盲区互补和 bi/cross-encoder 区别。
3. **微调（Q13）如实答，别夸大**——这是最容易翻车的点，讲清 RAG 和微调各解决什么反而加分。
4. 每个组件都要能回答『**为什么用它、不用会怎样**』——面试官考的是选型判断，不是名词。
5. **补齐真实量化**（Recall@k、faithfulness、各组件消融对比、延迟），个人项目没数据 = demo。



这句话在描述一个处理 PDF/文档类用户手册的技术方案,拆开看:

**PyMuPDF** — 一个 Python 库(导入名叫 `fitz`),用来从 PDF 里提取文字、图片、页面结构等。这里用它来"解析"用户手册,把 PDF 转成可处理的文本。

**滑动窗口(sliding window)** — 一种切分文本的方法。把长文本切成小块时,相邻块之间保留一部分重叠内容(比如每块 500 字,相邻两块重叠 100 字),避免在切分边界处把一句完整的话或一个语义单元切断,导致上下文丢失。

**父子文档(parent-child document)** — 一种分层的文本组织策略。检索时用**小块(子)**去匹配(更精准),但返回给模型的是包含它的**大块(父)**(上下文更完整)。兼顾检索精度和上下文丰富度,常用于 RAG(检索增强生成)。

**语义切分(semantic chunking)** — 按"语义/意思"来切,而不是机械地按固定字数切。通常用句向量判断相邻句子的语义相似度,在意思发生明显转折的地方切开,让每一块内部主题尽量一致。

**MongoDB** — 一个文档型数据库。这里用来存放切好的**文本块(chunks)**以及它们的**元数据**(比如来自哪一页、属于哪个父块、章节标题、块的编号等)。


这句话描述的是一套 **RAG 检索系统的召回+排序方案**,逐个拆解:

## 召回阶段(多路召回)

**召回(recall)** — 从海量文本块里先"粗略捞出"一批可能相关的候选。多路召回就是用几种不同原理的方法同时捞,再合并,提高覆盖率。

**Dense 语义召回 + Qwen3-Embedding** — "Dense(稠密)"指把文本转成一个稠密的向量(几百上千维、每维都是浮点数),用向量距离衡量语义相似度,能匹配"意思相近但用词不同"的内容。Qwen3-Embedding 是阿里通义千问团队出的一款 embedding(向量化)模型,负责把文本转成这种稠密向量。

**Sparse 语义召回 + BGE-M3** — "Sparse(稀疏)"指生成一个维度极高、但大部分为 0 的向量,每个非零维对应一个具体词及其权重,介于纯语义和纯字面之间。BGE-M3 是智源(BAAI)的多功能 embedding 模型,能同时产出稠密和稀疏表示,这里用它的稀疏能力。

**字面召回 + BM25** — BM25 是经典的关键词检索算法(基于词频、逆文档频率等),做的是**字面精确匹配**,擅长命中专有名词、型号、代码、精确术语这类"必须一字不差"的内容,弥补语义召回可能漏掉精确词的问题。

**混合召回策略** — 把上面三路(稠密语义、稀疏语义、字面)结果合在一起,兼顾"懂意思"和"抠字眼"。

**Milvus** — 一个开源的向量数据库,负责存储向量并高效执行向量相似度检索(即支撑 Dense/Sparse 召回的底层引擎)。

## 排序阶段

**RRF 做粗排** — RRF(Reciprocal Rank Fusion,倒数排名融合)是一种融合算法。多路召回各自给出一个排序列表,RRF 按每个结果在各列表里的**排名倒数**加权求和,把多个列表合并成一个统一的粗排序。它简单、无需训练、对多路结果融合效果好。

**BGE-Reranker 做精排** — 精排是对粗排后剩下的少量候选做更精细的重排。BGE-Reranker 是一个 **cross-encoder(交叉编码)重排模型**:它把"问题+候选文本"成对一起送进模型深度比对相关性,比召回阶段的向量相似度更准,但更慢,所以只用在少量候选上。

> 粗排(快、批量、降候选数)→ 精排(慢、精准、定最终顺序),是检索系统常见的两级漏斗结构。

## 问答输出能力

- **输出答案** — 基于检索到的手册内容生成回答。
- **引用页码** — 回答时标注信息来自手册的哪一页(依赖之前存进 MongoDB 的页码元数据),便于溯源核对。
- **关联插图** — 能把回答关联到手册里对应的图片/示意图一并呈现。

---

整体流程串起来:

**三路召回(Qwen3 稠密 + BGE-M3 稀疏 + BM25 字面,用 Milvus 检索)→ RRF 粗排融合 → BGE-Reranker 精排 → 生成带页码引用和关联插图的答案。**

这是一套面向"用户手册问答"的、比较完整且工程化的 RAG 检索链路,亮点在于多路混合召回保证召回全,两级排序保证结果准,输出层还兼顾了可溯源(页码)和多模态(插图)。
