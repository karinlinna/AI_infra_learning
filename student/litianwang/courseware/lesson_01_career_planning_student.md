## 6. 技术加餐 A：把 Tesla RAG 讲成"技术语言"（面试主力弹药）

### 6.1 一句话架构

> **"一个带页码溯源的车主手册 RAG 问答：PDF 解析 → 父子分块 → 三路混合召回（Dense + Sparse + BM25）→ RRF 粗排 → Reranker 精排 → LLM 生成并回引页码/插图。"**

把这句话拆成 6 个环节，每个环节你都要能答出「**做了什么 + 为什么这么做 + 生产用什么组件**」：

| 环节 | 做了什么 | 为什么（面试考点） | 生产组件 |
|---|---|---|---|
| **① 解析** | PyMuPDF 逐页抽文字 + 插图，记录页码 | 元数据（page/figure）是后面"引用溯源"的地基 | PyMuPDF(fitz) |
| **② 分块** | 父子文档：子块细粒度检索，命中回溯父块给全上下文 | 子块提召回精度，父块保上下文完整——解决"切太碎丢语义 / 切太大噪声多"的矛盾 | 语义切分 + 滑动窗口重叠 |
| **③ 召回** | Dense 稠密 + Sparse 稀疏 + BM25 字面，三路并行 | 单路都有盲区：BM25 只认字面、Dense 懂语义但可能漂——互补兜底 | Qwen3-Embedding / BGE-M3 |
| **④ 粗排** | RRF 把三路排名倒数相加融合 | **只看排名不看原始分**，天然免疫三路分数量纲不一致的问题 | Reciprocal Rank Fusion |
| **⑤ 精排** | Reranker 对 query-doc 逐对打分重排 | cross-encoder 比向量内积更准，用少量候选换高精度 | BGE-Reranker |
| **⑥ 生成** | 拼上下文喂 LLM，输出答案 + 引用页码 + 插图 | 可溯源 = 抗幻觉，是 RAG 落地的核心价值 | 微调后的 Qwen 等 |

### 6.2 三个必背的"技术为什么"（面试最爱追问）

**Q1｜为什么要多路召回，一路不行吗？**
> 单路都有盲区：BM25/Sparse 靠字面词命中，遇到"换个说法问"就失效；Dense 懂语义但可能语义漂移。三路并行让它们互相兜底。
> **杀手锏例子（一定要讲）**：项目里问"冬天开车前要做什么准备"，Sparse 和 BM25 都被"充电"这个词误召回到 p12（充电口），只有 Dense 稠密召回抓到 p93（低温注意事项），最后精排把 p93 排到第一。**这就是多路召回互补价值的活证据**——字面匹配失效时语义召回接住了。

**Q2｜为什么用 RRF 融合，不直接把三路分数加起来？**
> 三路分数量纲不一样（余弦相似度 0~1、BM25 可能是几十），直接相加会被大数值那路支配。RRF 只用「排名的倒数」`1/(k+rank)`，**只看谁排前面、不看原始分**，对不同尺度的召回天然鲁棒。`k`（常取 60）是平滑项，防止 rank=1 的文档权重过大。

**Q3｜父子分块解决什么问题？**
> 块切太碎 → 检索准但上下文残缺，LLM 答不全；切太大 → 上下文全但混入噪声、检索不准。父子结构两头都要：**用细粒度子块去检索（准），命中后回溯到父块喂给 LLM（全）**，一次拿到"检索精度"和"上下文完整"两个好处。

### 6.3 数据说话：把效果量化（面试官最想听的一句）

> 面试讲项目最忌"我做了 X"，要讲"我做了 X，指标从 A 提到 B"。哪怕是自测数据，也远胜没有数字。准备好这样一句：

> **"引入 Reranker 前，Top-5 命中率约 70%；加了 cross-encoder 精排后提到约 90%，人工抽查 50 个问题，答错从 12 个降到 3 个。"**

即使你没跑过严格评测，也要在下节课前**补一组自测数字**（哪怕人工标 30 题）——这是把项目从"听起来会"变成"确实做过"的分水岭。

## 8. 技术加餐 C：手写一个最小 Agent Loop（面试问"你懂 Agent 吗"的底气）

> 很多同学能说出"ReAct、工具调用"，但一被追问"循环里到底发生了什么"就卡住。**能手写 30 行核心循环，就把你和"只会调库的人"区分开了。** 这也是你第 2 节课要独立复现的东西。

### 8.1 Agent 的本质：一个"带工具的 while 循环"

去掉框架包装，Agent 就是让 LLM 反复做四件事，直到它认为任务完成：

```
┌─────────────────────────────────────────────┐
│  1. Think   模型根据当前上下文思考下一步       │
│  2. Act     模型决定：调用某个工具(带参数)     │
│  3. Observe 程序真正执行工具，把结果塞回上下文 │
│  4. Repeat  带着新结果回到第 1 步，直到给出答案 │
└─────────────────────────────────────────────┘
```

关键点：**LLM 本身不会执行任何动作**，它只是"输出想调哪个工具、传什么参数"这段文本/JSON；真正干活（查数据库、算数、发请求）的是你写的程序。这就是 Function Calling 的核心。

### 8.2 核心代码（Python 伪实现，能讲清就够）

```python
# 1) 定义工具：每个工具就是一个普通 Python 函数
def search_manual(query: str) -> str:
    """在车主手册里检索（这里可接你的 Tesla RAG 检索链路）"""
    return rag_retrieve(query)          # 复用你已有的 RAG 系统

def calculator(expr: str) -> str:
    return str(eval(expr))              # 教学用，生产要做安全校验

TOOLS = {"search_manual": search_manual, "calculator": calculator}

# 2) Agent 主循环
def run_agent(user_question: str, max_steps: int = 6) -> str:
    messages = [
        {"role": "system", "content": SYSTEM_PROMPT},   # 告诉模型有哪些工具、输出格式
        {"role": "user",   "content": user_question},
    ]

    for step in range(max_steps):                        # ← 防死循环的护栏
        # (1) Think + Act：让模型决定下一步
        resp = llm.chat(messages, tools=TOOL_SCHEMAS)

        # (2) 模型没要调工具 → 说明它给出了最终答案
        if not resp.tool_calls:
            return resp.content

        # (3) Observe：真正执行模型点名的工具
        for call in resp.tool_calls:
            fn = TOOLS[call.name]
            result = fn(**call.arguments)                # 程序执行，不是模型执行
            messages.append({"role": "assistant", "tool_call": call})
            messages.append({"role": "tool", "content": result})  # 结果塞回上下文

        # (4) 带着工具结果回到循环顶部，继续 Think

    return "达到最大步数仍未完成（可能任务过难或工具不够）"
```

### 8.3 这段代码里藏着的 5 个面试考点

| 代码位置 | 对应的面试考点 | 能这样答 |
|---|---|---|
| `max_steps` 护栏 | **Agent 怎么防死循环/失控？** | 设步数上限 + 超时 + 成本预算，是生产必备的安全阀 |
| `resp.tool_calls` 判空 | **Agent 怎么知道该停了？** | 模型不再请求工具、或触发终止条件时结束 |
| `fn(**call.arguments)` | **谁在执行动作？** | 模型只产出意图，执行由宿主程序做——这是安全边界 |
| 结果塞回 `messages` | **上下文怎么增长？** | 每轮把 observation 追加进去，所以要管 token/做裁剪 |
| `search_manual` 接 RAG | **RAG 和 Agent 什么关系？** | RAG 可以是 Agent 的一个工具——把你两个项目串成一条线讲 |

> 💡 **面试串讲金句**：*"我的 Tesla RAG 检索链路，可以直接封装成 Agent 的一个 `search_manual` 工具；Agent 负责多步规划和调度，RAG 负责单次高质量检索——这就是从 RAG 走到 Agent 的自然演进。"* 这句话能把你现有资产升维成"体系化理解"。

## 9. 技术加餐 D：RAG / Agent 面试高频追问（10 连问速答）

> 校招技术面常见的连环追问。每题先记**一句话核心答案**，再准备 30 秒展开。下节课我们挑几道现场演练。

**① Embedding 的相似度为什么常用余弦而不是欧氏距离？**
> 余弦只看方向不看模长，对文本长度不敏感；而 embedding 更关心语义方向。（很多向量库内部把向量归一化后，余弦=内积。）

**② 向量库为什么用 ANN 近似检索，不做精确最近邻？**
> 百万级向量精确暴力搜太慢。ANN（HNSW/IVF）用图或聚类索引，牺牲极小召回率换几十上百倍加速，是工程上的必然取舍。

**③ Chunk 大小怎么定？**
> 没有银弹，取决于文档结构和问题粒度。太小丢上下文、太大引噪声；实践中用父子/滑窗，并**用评估集调**——把 chunk_size 当超参数扫。

**④ 检索到的内容不相关（召回噪声）怎么办？**
> 三板斧：加 Reranker 精排过滤、提高 chunk 质量（语义切分）、召回后按分数阈值截断。再不行考虑 query 改写/扩展。

**⑤ 怎么减少 RAG 的幻觉？**
> ① 强约束 prompt："只依据给定资料回答，无依据就说不知道"；② 强制输出引用来源（你项目的页码溯源就是加分点）；③ 忠实度评估兜底。

**⑥ 怎么评估一个 RAG 系统好不好？**
> 拆成两段：**检索段**看 Recall@K / MRR；**生成段**看 Faithfulness（答案是否忠于检索内容）+ Answer Relevancy。可以用 RAGAS 这类框架。

**⑦ Function Calling 和直接让模型输出 JSON 有什么区别？**
> 本质都是结构化输出。Function Calling 是模型原生支持的、更可靠的机制，能返回规范的工具名+参数；纯 prompt 让它吐 JSON 容易格式跑偏，需要额外校验/重试。

**⑧ Agent 多步任务中间错了一步，怎么恢复？**
> 把工具报错作为 observation 喂回去，让模型看到错误自行纠偏（反思/重试）；配合步数上限和关键步骤的人工确认（human-in-the-loop）。

**⑨ 长对话上下文超了怎么办？**
> 滑动窗口保留最近 N 轮 + 把更早内容摘要压缩 + 需要时从长期记忆（向量库）检索回填。核心是"上下文是有限资源要主动管理"。

**⑩ RAG 和微调（Fine-tuning）什么时候用哪个？**
> RAG 解决"知识新/多/要溯源"的问题，改数据不改模型；微调解决"风格/格式/领域能力"的问题，改模型行为。**常见组合**：微调调语气和输出格式，RAG 供事实知识——不是二选一。
